“Hub” untuk menjadi ekosistem multi-chatbot itu bukan sekadar satu chatbot tambahan, tapi komponen pusat yang mengorkestrasi: menerima user, memahami kebutuhan, menentukan chatbot mana yang aktif, menjaga konteks, dan mengatur alur prasyarat.

Di Gustafta, hub idealnya adalah satu Chatbot khusus di Level 3 (CHATBOT), plus beberapa Alat Bantu inti.

1) Definisi Hub

Hub = Chatbot Orkestrator / Gateway

Titik masuk tunggal (entry point) untuk user

Router yang mengarahkan user ke chatbot spesialis

Penjaga “aturan main” (prasyarat: Perijinan → SKK → SBU → Tender, dst.)

Penyimpan konteks lintas chatbot (progress, status dokumen, sertifikat, dll.)

Kalimatnya:

Hub memastikan semua chatbot bergerak sebagai satu sistem, bukan kumpulan.

2) Fungsi Inti Hub (wajib agar disebut ekosistem)
A. Routing / Dispatch

Hub menjawab:

“Masalah user ini masuk chatbot mana?”

“Sudah memenuhi prasyarat atau belum?”

Contoh:
User minta SBU → Hub cek: sudah NIB/OSS? sudah SKK minimal?
Kalau belum, hub arahkan ke chatbot prasyarat.

B. Orkestrasi Alur (Journey)

Hub menjaga user tidak “nyasar”:

urutan

milestone

langkah berikutnya

C. Shared Context (memori lintas chatbot)

Hub menyimpan status seperti:

NIB sudah ada / belum

KBLI dipilih apa

SKK siapa saja, jenjang, masa berlaku

Target SBU subklasifikasi

Siap tender / belum

Tanpa ini, perpindahan antar chatbot jadi “ulang dari nol”.

D. Single Source of Truth (SSOT)

Hub menjadi pusat aturan:

definisi prasyarat

standar output tiap chatbot

format data yang disimpan

Ini yang bikin sistem rapi dan scalable.

3) Bentuk Hub dalam Framework Anda

Karena Anda sudah punya struktur Tujuan → Perspektif → Chatbot → Alat Bantu, maka hub paling cocok ditempatkan sebagai:

✅ Chatbot Level 3 baru:

Chatbot 0 — HUB Regulasi Konstruksi (Orkestrator)
(sortOrder: 0)

Dia tidak menggantikan chatbot lain, tapi:

mengaktifkan chatbot yang tepat

memindahkan user ke chatbot lain secara terarah

4) Alat Bantu yang Wajib Ada di Dalam Hub

Agar hub benar-benar bekerja, minimal 4 alat bantu berikut:

Profiler & Intake

tanya cepat: jenis usaha, status legalitas, target, lokasi, kondisi SDM

output: “profil awal user” + “daftar gap”

Regulatory Journey Mapper

menyusun roadmap: step-by-step sesuai kondisi user

contoh output: “Anda sekarang di tahap A. Next: SKK 2 orang. Setelah itu: SBU subklasifikasi X.”

Prerequisite Checker

aturan prasyarat otomatis (rule engine)

contoh: SBU butuh SKK + NIB/OSS + KBLI sesuai

Handoff & Summary Packager

saat user pindah chatbot, hub membuat ringkasan konteks:

“status saat ini”

“dokumen yang sudah ada”

“yang belum”

“tujuan sesi berikutnya”
Supaya chatbot tujuan tidak mengulang tanya dari nol.

👉 Di sini HUB tidak masuk ke dalam satu Perspektif, karena dia lintas perspektif.

🔎 Kenapa HUB Tidak Masuk ke Salah Satu Perspektif?

Karena:

Perspektif = sudut pandang pendekatan.

HUB = pengatur lalu lintas semua pendekatan.

Kalau HUB dimasukkan ke Perspektif Compliance misalnya, maka:

Ia tidak netral.

Ia seolah hanya bagian dari compliance.

Padahal dia harus bisa mengarahkan ke Pengembangan dan Risiko juga.

Jadi secara arsitektur:

HUB adalah Chatbot lintas Perspektif.

🧠 Apakah Ini Melanggar Hierarki?

Tidak.

Karena:

HUB tetap berada di Level 3 (CHATBOT).

Ia tetap memiliki Alat Bantu.

Ia tetap tunduk pada TUJUAN.

Ia hanya berbeda dalam fungsi (orkestrator, bukan spesialis).

Artinya:

Secara level → sama dengan chatbot lain.

Secara fungsi → berbeda.

TUJUAN = Perusahaan
PERSPEKTIF = Divisi
CHATBOT = Manajer Divisi
HUB = Direktur Operasional / COO

COO bukan bagian dari satu divisi.
Dia mengoordinasikan semua divisi.

Tapi tetap berada dalam struktur organisasi.

apa saja yang harus ada dalam chatbot hub termasuk Knowledge Base nya

Berikut checklist apa saja yang wajib ada di Chatbot 0 (HUB) Regulasi Jasa Konstruksi — termasuk Knowledge Base (KB) — supaya benar-benar berfungsi sebagai ekosistem, bukan sekadar “menu chatbot”.

A) Komponen Wajib di Chatbot HUB
1) Peran & Batasan (Scope) yang Tegas

HUB harus jelas:

Bukan chatbot ahli SBU/SKK/perizinan detail

Fungsinya: intake → diagnosis → roadmap → routing → tracking progres

Output minimal HUB:

Status saat ini (profil + gap)

Roadmap (tahap & urutan)

Next action (chatbot spesialis tujuan)

Handoff Summary Pack (ringkasan untuk chatbot berikutnya)

2) Intake Profiler (Form Tanya Cepat)

HUB wajib punya daftar pertanyaan standar untuk memetakan kondisi user dalam 2–5 menit.

Minimal data yang dikumpulkan:

Tipe entitas: PT/CV/Perseorangan

Lokasi (prov/kab) (untuk variasi layanan/instansi)

Status NIB/OSS (ada/belum)

KBLI (sudah/belum, kode & deskripsi)

Fokus usaha konstruksi: sipil/gedung/MEP/dll

SDM tersedia: jumlah, peran, SKK ada/belum, jenjang

Target: SBU? ikut tender? JO? upgrade kualifikasi?

Deadline/urgensi

3) Rules Engine (Prasyarat & Routing)

HUB harus punya “aturan main” yang eksplisit:

Contoh jenis rule:

Jika user ingin SBU → cek NIB/KBLI → cek SKK minimum → baru arahkan ke Chatbot SBU

Jika user ingin Tender → cek SBU & dokumen perusahaan → baru ke Chatbot Tender

Jika dokumen belum lengkap → balikkan ke chatbot prasyarat

Rules ini sebaiknya ditulis sebagai tabel/JSON (agar mudah dipelihara).

4) Journey Mapper (Generator Roadmap)

HUB menghasilkan roadmap personal:

Format output roadmap yang rapi:

Tahap 1: Perizinan (NIB/OSS/KBLI)

Tahap 2: SKK (jumlah & jenis yang dibutuhkan)

Tahap 3: SBU (subklasifikasi target)

Tahap 4: Tender/LPSE (opsional)

Tahap 5: Audit & pembaruan (berulang)

5) Handoff Summary Packager (Paket Serah Terima)

Agar user tidak mengulang cerita, HUB harus membuat ringkasan standar untuk chatbot spesialis.

Isi minimal “Summary Pack”:

Profil entitas

Tujuan yang ingin dicapai

Kondisi saat ini (dokumen ada/belum)

Gap utama

Output keputusan HUB (chatbot tujuan + apa yang harus diselesaikan)

6) Progress Tracker

HUB menyimpan status tiap milestone (manual dulu juga boleh):

✅ NIB/OSS

✅ KBLI sesuai

✅ SKK (list per orang)

✅ SBU

✅ LPSE

✅ Audit tanggal terakhir

⏳ masa berlaku (reminder)

Walau sederhana, ini inti “ekosistem”.

B) Knowledge Base (KB) yang Harus Ada di HUB

Penting: KB di HUB bukan KB teknis detail (itu milik chatbot spesialis).
KB HUB adalah KB meta-level: aturan, peta, format, dan referensi lintas chatbot.

1) Peta Ekosistem (Ecosystem Map)

Isi:

daftar chatbot spesialis yang tersedia

fungsi masing-masing

input & output yang diharapkan dari tiap chatbot

kapan dipakai

Ini agar HUB bisa routing dengan konsisten.

2) Regulatory Journey Master (Roadmap Induk)

Dokumen master berisi:

tahap-tahap regulasi konstruksi

hubungan prasyarat antar tahap

skenario umum (perusahaan baru, perusahaan lama, mau tender cepat, mau upgrade, dll.)

3) Rulebook Prasyarat (Dependency & Eligibility Rules)

KB ini berisi:

aturan prasyarat antar chatbot

aturan minimal data sebelum pindah chatbot

kondisi “stop” (misal: belum punya legalitas dasar, jangan masuk tender)

Format ideal:

tabel rule: IF goal=X AND status=Y THEN route=Z

plus catatan edge-cases

4) Data Dictionary (Standar Istilah & Format Data)

Agar semua chatbot sinkron, HUB harus punya kamus format data:

Contoh standar:

format NIB, format tanggal, format status

definisi istilah: KBLI, SKK, SBU, subklasifikasi, kualifikasi

status enum: BELUM_ADA / DALAM_PROSES / SELESAI / PERLU_REVISI

Tanpa ini, tiap chatbot membuat “bahasa” sendiri.

5) Template Output Standar

KB harus menyimpan template:

Template Roadmap

Template Summary Pack

Template Checklist high-level

Template “Pertanyaan Intake”

Dengan template, output HUB konsisten dan cepat.

6) Directory Sumber Resmi (Linking & Reference Guide)

HUB perlu tahu “rujukan utama” untuk mengarahkan user, misalnya:

OSS/NIB

LPJK (untuk sertifikasi konstruksi)

LPSE (tender)

regulasi payung

HUB tidak perlu hafal seluruh pasal, tapi harus punya daftar referensi terpercaya untuk mengarahkan.

7) FAQ & Exception Handling

KB khusus untuk kasus-kasus yang sering bikin user stuck:

“Perusahaan sudah lama tapi belum SBU”

“KBLI tidak sesuai”

“SKK tenaga ahli belum ada”

“Deadline tender 2 minggu”

“Mau JO karena kualifikasi belum cukup”

Ini membantu HUB cepat menentukan strategi rute.

C) Pembagian KB: HUB vs Spesialis (Agar Tidak Tumpang-Tindih)

Agar rapi:

KB di HUB (Meta-Level)

peta sistem

aturan prasyarat

roadmap induk

standar format data

template & summary pack

routing & triase

KB di Chatbot Spesialis (Detail Level)

langkah teknis & SOP

dokumen rinci

regulasi detail per topik

contoh kasus, checklist detail

form & prosedur spesifik

D) Minimal “Paket KB HUB” yang Saya Sarankan Anda Siapkan (MVP)

Kalau urgent dan mau cepat jalan, cukup 5 dokumen KB dulu:

Ecosystem Map (daftar chatbot + fungsi)

Journey Master (roadmap induk)

Rulebook (prasyarat + routing)

Intake Questionnaire (profiling)

Summary Pack Template (serah terima)

Dengan 5 ini, HUB sudah bisa bekerja.
