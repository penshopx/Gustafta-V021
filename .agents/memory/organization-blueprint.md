# AI Organization Blueprint â€” design contract

Fase 3 extends the no-code Dialog â†’ Blueprint â†’ Configuration flow so it can design a whole **organization of agents** (orchestrator + specialists that collaborate), not just one chatbot. Still 100% no-code for the user; the user only ever uses dialog/blueprint/configuration.

Foundation = `shared/blueprint/organization-blueprint-schema.ts` (`OrganizationBlueprint`), pure/additive (Tahap 18), not imported by app code yet. Roadmap: `docs/blueprint-engine/00-roadmap.md` (Fase 3 table), detail `docs/blueprint-engine/18-organization-blueprint-schema.md`.

## Shape
- `meta` â€” name/mission/intent/status + `agentIds` (localIdâ†’agentId, filled LATER by config engine).
- `members[]` â€” each seat = `{ localId, role: orchestrator|specialist|support, title?, responsibility?, blueprint }`. **Reuses the single-agent `blueprintSchema`** per member (no field duplication).
- `structure` â€” `{ leadLocalId?, edges[] }`; directed graph `fromLocalIdâ†’toLocalId` that becomes each orchestrator's `agenticSubAgents` after resolution.

## Non-obvious decisions (must hold across later stages)
- **localId, NOT agentId, at design time.** Agents have no DB id until creation. Members reference each other by `localId`; the Configuration Engine resolves `localIdâ†’agentId` AFTER creating agents, then writes `agenticSubAgents`. **Why:** edges need an agentId that doesn't exist yet (chicken-and-egg).
- **Single source of truth for wiring = org `structure.edges`**, NOT member-level `orchestration.agenticSubAgents` / `meta.agentId` / `parentAgentId`. `lintOrganizationBlueprint` WARNS if a member sets those. **How to apply:** the future Mapping Engine (Tahap 19) MUST treat org edges as authoritative and ignore/normalize member-level runtime linkage fields â€” otherwise dual source of truth.
- **Lint warns, never throws** â€” org is partial-by-design during dialogue.
- **`role` is authoritative for topology, not the member's blueprint.** The Mapping Engine (Tahap 19) forces `agentPatch.isOrchestrator = (role === "orchestrator")` for EVERY member â€” never just sets true for orchestrators. **Why:** a specialist/support member whose embedded single-agent blueprint carries `isOrchestrator:true` would otherwise leak through `mapBlueprintToBuilder`'s generic 1:1 field copy â†’ a second source of truth for team topology. Same family of bug as the member-level edge fields.

## Tahap 19 â€” Organization Mapping Engine (done)
`server/services/blueprint-engine/organization-mapping-engine.ts` `mapOrganizationToBuilder(org, options?)` is pure: per member calls the existing single-agent `mapBlueprintToBuilder` (no duplicated logic; it returns a fresh object per call so mutating its `agentPatch` is safe), strips forbidden member wiring keys `["agenticSubAgents","parentAgentId"]`, forces `isOrchestrator` from role, and builds `wiring[]` (orchestratorâ†’sub-agents) ONLY from `structure.edges`, still by `localId`. Edge handling: self-edges, dangling refs, and edges whose source isn't an orchestrator are skipped; duplicate sub-agent edges dropped with warning; orchestrators with no outgoing edge warned. `localIdâ†’agentId` resolution is deliberately deferred to Tahap 20.

## Tahap 20 â€” Organization Configuration Engine (done)
`server/services/blueprint-engine/organization-configuration-engine.ts` `applyOrganizationToBuilder(org, { storage, ownerUserId, dryRun })`. Validate-all-before-write; `dryRun` previews only.
- **Atomicity is mandatory, not best-effort.** Real (non-dryRun) writes REFUSE to run unless `storage.runInTransaction` exists â€” it does NOT fall back to non-transactional writes, because a partial team is worse than no team. Tahap 21 must implement `runInTransaction` on the real `DbStorage` (db.transaction pattern) before exposing org writes.
- **Two-phase inside the transaction.** Phase A: create every agent (+ its children) recording `localIdâ†’agentId`, deferring `isOrchestrator` (insertAgentSchema's refine requires bigIdeaId when isOrchestrator true). Phase B: set `isOrchestrator`/`orchestratorRole`/`agenticSubAgents` from the resolved wiring; coerce `agenticSubAgents.agentId` to numeric.
- **Children must be written `strict` so a child failure rolls back the whole org.** `applyChildren` (in single-agent `configuration-engine.ts`) gained a `strict=false` param: in strict mode any child validation OR insert failure THROWS instead of becoming a warning. The org engine calls it with `strict=true`; single-agent Tahap 4 path keeps default false (best-effort). **Why:** without this, a child insert failure inside the transaction was swallowed â†’ tx committed a partial org while returning applied:true. **How to apply:** any future transactional batch reusing `applyChildren` MUST pass strict=true; treat default-false as the documented single-agent invariant.
- Stamp `ownerUserId` on EVERY created agent (same trap as single-agent `/configure` create â€” agent vanishes from owner's dashboard if userId not stamped post-parse).

## Tahap 21 â€” API + transactional storage wiring (done)
- **`runInTransaction` is implemented ONLY on the real `DatabaseStorage`, NOT on `IStorage`.** Putting it on the interface would force `MemStorage` to implement it. The org configuration engine uses the singleton storage's method at runtime; tests use an in-memory fake. **How to apply:** if a future stage needs the in-memory storage to honor transactions, add it as an optional capability check, don't widen `IStorage`.
- **How tx-scoping reuses existing write logic without duplication:** 6 create/update methods (+ private `lookupSeriesNameForAgent`) gained an OPTIONAL trailing `exec: Executor = db` param (`Executor = typeof db | tx-type`). All existing callers pass nothing â†’ identical behavior. `runInTransaction(fn)` builds a tx-scoped `ConfigStorage` whose create/update delegate to `this.<m>(args, tx)`, runs `db.transaction`, clears agent caches AFTER commit. **Why:** lets one transaction reuse the exact create/update code path instead of cloning insert logic.
- **Cache writes inside those methods are guarded `if (exec === db)`.** When running inside a transaction, the in-memory caches (agentCache/agentListCache/kbCache) are NOT touched â€” uncommitted rows must never populate a cache that a concurrent request could read. **How to apply:** any further method made tx-aware MUST guard its cache mutations the same way, or a rolled-back row leaks into cache.
- **Routes:** `server/organization-engine-routes.ts` mirrors `blueprint-engine-routes.ts` (HttpError/handler/sessionUserId). Org engine is CREATE-ONLY (no mode/update param). `POST /api/organization/analyze` = read-only lint+plan summary; `POST /api/organization/configure` = write, `isAuthenticated`, `dryRun` DEFAULT true (must send `dryRun:false`), stamps `ownerUserId` from session. Registered next to blueprint routes in `server/routes.ts`.
- Full `npm run check` / single-file `tsc --noEmit` get KILLED (too heavy); verify via `npx tsx --test` (compiles on the fly) + server boot logs + curl 401 probe instead.

## Tahap 22 â€” UI Wizard Organisasi (done)
`client/src/pages/organization-builder.tsx` (route `/organization-builder`, auth-gated; entry card "Rakit Tim AI" di Aksi Cepat Dashboard). Mirror of single-agent `blueprint-builder.tsx` but builds the `OrganizationBlueprint` ENTIRELY client-side via `createEmptyOrganizationBlueprint` + per-member `createEmptyBlueprint`, then calls Tahap 21 routes. Steps introâ†’membersâ†’reviewâ†’done.
- **Member blueprint must NOT carry topology.** The wizard sets only `meta.intent`, `modules.identity.data.name/description`, `modules.aiEngine.data.systemPrompt` per member; role/wiring live on the org member `role` + `structure.edges` (single source of truth â€” see Tahap 19). Setting orchestration in the member blueprint would be stripped/overridden anyway. **Why:** dual source of truth bug.
- **Wiring is generated, not asked.** Edges = the single lead orchestrator â†’ every other member; `leadLocalId` set. UI never lets the user hand-draw edges.
- **Topology hard-gate in UI (not just lint).** `topologyValid = exactly 1 orchestrator && members.length >= 2`. Both analyze and create are disabled until valid, with an inline `topologyHint`. **Why:** the org engine + lint are permissive (warn, not throw), so without a UI gate a non-technical user could silently materialize a disconnected or multi-orchestrator team.
- **`applied:false` needs its own error state, separate from the dryRun preview panel.** A dryRun preview returns `applied:false` as NORMAL success, so you cannot key the failure UI on `applied`. Real create failure sets a dedicated `createError` string[] rendered as a red banner; never reuse the green "Pratinjau berhasil" panel for failures (architect Tahap 22 finding).
- **Empty systemPrompt â†’ auto-generated default** (from member title + responsibility + org mission) so every created agent has a working persona, never a blank brain.
- minConfidence defaults to 0 in mapping-engine â†’ UI-filled fields (no fieldMeta) DO map to agentPatch; no need to attach confidence metadata client-side.

## Tahap 23 â€” Organization Dialogue / Inference Engine (done)
`server/services/blueprint-engine/organization-dialogue-engine.ts` â€” PURE (no DB/LLM/UI), org-level mirror of single-agent Dialogue (T5) + Inference (T6). NOT yet wired to any route/UI (that's a later tahap). Tests: `tests/organization-dialogue-engine.test.ts` (18).
- **Per-member field inference REUSES the exported single-agent `inferBlueprint`, never reimplements it.** Only the domainâ†’team data (`TEAM_TEMPLATES`: which seats to propose per domain) is new â€” that is legitimately org-specific composition knowledge, not duplicated inference logic. **How to apply:** if you need richer per-agent inference inside an org, extend the single-agent inference rules, not the org engine.
- **`inferOrganization` MUST call `stripMemberWiring` after `inferBlueprint`.** A reused single-agent inference can leave `orchestration.*`/`meta.agentId` shaped data on a member; the wiring source-of-truth is `structure.edges` ONLY (see Tahap 18/19). Strip `agenticSubAgents`/`parentAgentId`/`meta.agentId` from every member or lint breaks. **Why:** dual source of truth for topology.
- **Every exported fn returns a FRESH clone in ALL branches â€” including no-op/failure paths.** `applyOrgAnswer` clones FIRST then validates (early returns hand back the clone, not the input); `applyOrgAnswers` seeds `current = cloneOrg(org)` so even an empty answers map returns a new object. **Why:** architect flagged the original returning the input ref on invalid input. Note the single-agent `applyAnswer` does NOT do this (returns input on failure) â€” the org engine is intentionally stricter; don't "fix" it back.
- **`overallConfidence` = round2(0.5Â·avgMemberReadiness + 0.5Â·structureScore).** memberReadiness âˆˆ[0,1] (name 0.34 + systemPrompt 0.33 + intent/desc 0.33); structureScore âˆˆ[0.25,1] (needs exactly 1 orchestrator + â‰¥1 other + lead-reachable fraction). Auto-structure only fires when `edges.length===0 && members>=2` and lead must be an orchestrator (leadLocalId pointing at a non-orchestrator is ignored).
- **Guided team design entry point = `suggestTeamComposition(missionText)` â†’ `applyTeamSuggestion(org, suggestion)`.** suggest is deterministic (keyword domain match, falls back to GENERIC_TEAM "Umum"); apply REPLACES members+structure (preserves meta), index 0 always the orchestrator lead, edges auto-drawn leadâ†’others.
