# AI Organization Blueprint тАФ design contract

Fase 3 extends the no-code Dialog тЖТ Blueprint тЖТ Configuration flow so it can design a whole **organization of agents** (orchestrator + specialists that collaborate), not just one chatbot. Still 100% no-code for the user; the user only ever uses dialog/blueprint/configuration.

Foundation = `shared/blueprint/organization-blueprint-schema.ts` (`OrganizationBlueprint`), pure/additive (Tahap 18), not imported by app code yet. Roadmap: `docs/blueprint-engine/00-roadmap.md` (Fase 3 table), detail `docs/blueprint-engine/18-organization-blueprint-schema.md`.

## Shape
- `meta` тАФ name/mission/intent/status + `agentIds` (localIdтЖТagentId, filled LATER by config engine).
- `members[]` тАФ each seat = `{ localId, role: orchestrator|specialist|support, title?, responsibility?, blueprint }`. **Reuses the single-agent `blueprintSchema`** per member (no field duplication).
- `structure` тАФ `{ leadLocalId?, edges[] }`; directed graph `fromLocalIdтЖТtoLocalId` that becomes each orchestrator's `agenticSubAgents` after resolution.

## Non-obvious decisions (must hold across later stages)
- **localId, NOT agentId, at design time.** Agents have no DB id until creation. Members reference each other by `localId`; the Configuration Engine resolves `localIdтЖТagentId` AFTER creating agents, then writes `agenticSubAgents`. **Why:** edges need an agentId that doesn't exist yet (chicken-and-egg).
- **Single source of truth for wiring = org `structure.edges`**, NOT member-level `orchestration.agenticSubAgents` / `meta.agentId` / `parentAgentId`. `lintOrganizationBlueprint` WARNS if a member sets those. **How to apply:** the future Mapping Engine (Tahap 19) MUST treat org edges as authoritative and ignore/normalize member-level runtime linkage fields тАФ otherwise dual source of truth.
- **Lint warns, never throws** тАФ org is partial-by-design during dialogue.
- **`role` is authoritative for topology, not the member's blueprint.** The Mapping Engine (Tahap 19) forces `agentPatch.isOrchestrator = (role === "orchestrator")` for EVERY member тАФ never just sets true for orchestrators. **Why:** a specialist/support member whose embedded single-agent blueprint carries `isOrchestrator:true` would otherwise leak through `mapBlueprintToBuilder`'s generic 1:1 field copy тЖТ a second source of truth for team topology. Same family of bug as the member-level edge fields.

## Tahap 19 тАФ Organization Mapping Engine (done)
`server/services/blueprint-engine/organization-mapping-engine.ts` `mapOrganizationToBuilder(org, options?)` is pure: per member calls the existing single-agent `mapBlueprintToBuilder` (no duplicated logic; it returns a fresh object per call so mutating its `agentPatch` is safe), strips forbidden member wiring keys `["agenticSubAgents","parentAgentId"]`, forces `isOrchestrator` from role, and builds `wiring[]` (orchestratorтЖТsub-agents) ONLY from `structure.edges`, still by `localId`. Edge handling: self-edges, dangling refs, and edges whose source isn't an orchestrator are skipped; duplicate sub-agent edges dropped with warning; orchestrators with no outgoing edge warned. `localIdтЖТagentId` resolution is deliberately deferred to Tahap 20.

## Tahap 20 тАФ Organization Configuration Engine (done)
`server/services/blueprint-engine/organization-configuration-engine.ts` `applyOrganizationToBuilder(org, { storage, ownerUserId, dryRun })`. Validate-all-before-write; `dryRun` previews only.
- **Atomicity is mandatory, not best-effort.** Real (non-dryRun) writes REFUSE to run unless `storage.runInTransaction` exists тАФ it does NOT fall back to non-transactional writes, because a partial team is worse than no team. Tahap 21 must implement `runInTransaction` on the real `DbStorage` (db.transaction pattern) before exposing org writes.
- **Two-phase inside the transaction.** Phase A: create every agent (+ its children) recording `localIdтЖТagentId`, deferring `isOrchestrator` (insertAgentSchema's refine requires bigIdeaId when isOrchestrator true). Phase B: set `isOrchestrator`/`orchestratorRole`/`agenticSubAgents` from the resolved wiring; coerce `agenticSubAgents.agentId` to numeric.
- **Children must be written `strict` so a child failure rolls back the whole org.** `applyChildren` (in single-agent `configuration-engine.ts`) gained a `strict=false` param: in strict mode any child validation OR insert failure THROWS instead of becoming a warning. The org engine calls it with `strict=true`; single-agent Tahap 4 path keeps default false (best-effort). **Why:** without this, a child insert failure inside the transaction was swallowed тЖТ tx committed a partial org while returning applied:true. **How to apply:** any future transactional batch reusing `applyChildren` MUST pass strict=true; treat default-false as the documented single-agent invariant.
- Stamp `ownerUserId` on EVERY created agent (same trap as single-agent `/configure` create тАФ agent vanishes from owner's dashboard if userId not stamped post-parse).

## Tahap 21 тАФ API + transactional storage wiring (done)
- **`runInTransaction` is implemented ONLY on the real `DatabaseStorage`, NOT on `IStorage`.** Putting it on the interface would force `MemStorage` to implement it. The org configuration engine uses the singleton storage's method at runtime; tests use an in-memory fake. **How to apply:** if a future stage needs the in-memory storage to honor transactions, add it as an optional capability check, don't widen `IStorage`.
- **How tx-scoping reuses existing write logic without duplication:** 6 create/update methods (+ private `lookupSeriesNameForAgent`) gained an OPTIONAL trailing `exec: Executor = db` param (`Executor = typeof db | tx-type`). All existing callers pass nothing тЖТ identical behavior. `runInTransaction(fn)` builds a tx-scoped `ConfigStorage` whose create/update delegate to `this.<m>(args, tx)`, runs `db.transaction`, clears agent caches AFTER commit. **Why:** lets one transaction reuse the exact create/update code path instead of cloning insert logic.
- **Cache writes inside those methods are guarded `if (exec === db)`.** When running inside a transaction, the in-memory caches (agentCache/agentListCache/kbCache) are NOT touched тАФ uncommitted rows must never populate a cache that a concurrent request could read. **How to apply:** any further method made tx-aware MUST guard its cache mutations the same way, or a rolled-back row leaks into cache.
- **Routes:** `server/organization-engine-routes.ts` mirrors `blueprint-engine-routes.ts` (HttpError/handler/sessionUserId). Org engine is CREATE-ONLY (no mode/update param). `POST /api/organization/analyze` = read-only lint+plan summary; `POST /api/organization/configure` = write, `isAuthenticated`, `dryRun` DEFAULT true (must send `dryRun:false`), stamps `ownerUserId` from session. Registered next to blueprint routes in `server/routes.ts`.
- Full `npm run check` / single-file `tsc --noEmit` get KILLED (too heavy); verify via `npx tsx --test` (compiles on the fly) + server boot logs + curl 401 probe instead.

## Tahap 22 тАФ UI Wizard Organisasi (done)
`client/src/pages/organization-builder.tsx` (route `/organization-builder`, auth-gated; entry card "Rakit Tim AI" di Aksi Cepat Dashboard). Mirror of single-agent `blueprint-builder.tsx` but builds the `OrganizationBlueprint` ENTIRELY client-side via `createEmptyOrganizationBlueprint` + per-member `createEmptyBlueprint`, then calls Tahap 21 routes. Steps introтЖТmembersтЖТreviewтЖТdone.
- **Member blueprint must NOT carry topology.** The wizard sets only `meta.intent`, `modules.identity.data.name/description`, `modules.aiEngine.data.systemPrompt` per member; role/wiring live on the org member `role` + `structure.edges` (single source of truth тАФ see Tahap 19). Setting orchestration in the member blueprint would be stripped/overridden anyway. **Why:** dual source of truth bug.
- **Wiring is generated, not asked.** Edges = the single lead orchestrator тЖТ every other member; `leadLocalId` set. UI never lets the user hand-draw edges.
- **Topology hard-gate in UI (not just lint).** `topologyValid = exactly 1 orchestrator && members.length >= 2`. Both analyze and create are disabled until valid, with an inline `topologyHint`. **Why:** the org engine + lint are permissive (warn, not throw), so without a UI gate a non-technical user could silently materialize a disconnected or multi-orchestrator team.
- **`applied:false` needs its own error state, separate from the dryRun preview panel.** A dryRun preview returns `applied:false` as NORMAL success, so you cannot key the failure UI on `applied`. Real create failure sets a dedicated `createError` string[] rendered as a red banner; never reuse the green "Pratinjau berhasil" panel for failures (architect Tahap 22 finding).
- **Empty systemPrompt тЖТ auto-generated default** (from member title + responsibility + org mission) so every created agent has a working persona, never a blank brain.
- minConfidence defaults to 0 in mapping-engine тЖТ UI-filled fields (no fieldMeta) DO map to agentPatch; no need to attach confidence metadata client-side.

## Tahap 23 тАФ Organization Dialogue / Inference Engine (done)
`server/services/blueprint-engine/organization-dialogue-engine.ts` тАФ PURE (no DB/LLM/UI), org-level mirror of single-agent Dialogue (T5) + Inference (T6). NOT yet wired to any route/UI (that's a later tahap). Tests: `tests/organization-dialogue-engine.test.ts` (18).
- **Per-member field inference REUSES the exported single-agent `inferBlueprint`, never reimplements it.** Only the domainтЖТteam data (`TEAM_TEMPLATES`: which seats to propose per domain) is new тАФ that is legitimately org-specific composition knowledge, not duplicated inference logic. **How to apply:** if you need richer per-agent inference inside an org, extend the single-agent inference rules, not the org engine.
- **`inferOrganization` MUST call `stripMemberWiring` after `inferBlueprint`.** A reused single-agent inference can leave `orchestration.*`/`meta.agentId` shaped data on a member; the wiring source-of-truth is `structure.edges` ONLY (see Tahap 18/19). Strip `agenticSubAgents`/`parentAgentId`/`meta.agentId` from every member or lint breaks. **Why:** dual source of truth for topology.
- **Every exported fn returns a FRESH clone in ALL branches тАФ including no-op/failure paths.** `applyOrgAnswer` clones FIRST then validates (early returns hand back the clone, not the input); `applyOrgAnswers` seeds `current = cloneOrg(org)` so even an empty answers map returns a new object. **Why:** architect flagged the original returning the input ref on invalid input. Note the single-agent `applyAnswer` does NOT do this (returns input on failure) тАФ the org engine is intentionally stricter; don't "fix" it back.
- **`overallConfidence` = round2(0.5┬╖avgMemberReadiness + 0.5┬╖structureScore).** memberReadiness тИИ[0,1] (name 0.34 + systemPrompt 0.33 + intent/desc 0.33); structureScore тИИ[0.25,1] (needs exactly 1 orchestrator + тЙе1 other + lead-reachable fraction). Auto-structure only fires when `edges.length===0 && members>=2` and lead must be an orchestrator (leadLocalId pointing at a non-orchestrator is ignored).
- **Guided team design entry point = `suggestTeamComposition(missionText)` тЖТ `applyTeamSuggestion(org, suggestion)`.** suggest is deterministic (keyword domain match, falls back to GENERIC_TEAM "Umum"); apply REPLACES members+structure (preserves meta), index 0 always the orchestrator lead, edges auto-drawn leadтЖТothers. This is what a future "susun otomatis dari misi" button in the wizard should call.

- **Route baru = `POST /api/organization/suggest` (read-only, auth-gated).** Sengaja TIDAK membangun OrganizationBlueprint penuh di server тАФ ia cuma mengembalikan `{ domain, members }` dari `suggestTeamComposition(mission)`. - **Auto-compose hanya PRE-FILL draft anggota, lalu pengguna tetap lewat alur lama** (membersтЖТanalyzeтЖТpreviewтЖТcreate). Tidak ada jalur "compose тЖТ langsung create". **Why:** topology gate + applied:false error state + dryRun preview semua sudah hidup di alur lama; melewatinya = lubang.

## Tahap 29 тАФ Auto-simpan rancangan tim (draft localStorage) (done)
Wizard `/organization-builder` menyimpan draft (nama, misi, anggota, maxSpecialists) ke localStorage `gustafta_org_builder_draft_v1`; saat dibuka lagi tawarkan banner "Lanjutkan / Mulai baru". Murni klien.
- **Pola gating dua-ref untuk auto-save + restore-banner.** `bootRef` (sudah baca storage) + `holdRef` (tahan persist selama menunggu keputusan restore). **Why:** persist effect & mount-load effect jalan di commit yang sama; tanpa `holdRef` yang di-set DI DALAM mount effect (deklarasi SEBELUM persist effect), persist effect (closure restorable=null) akan menghapus draft dari storage sebelum user menekan "Lanjutkan". **How to apply:** mount-load effect HARUS dideklarasikan sebelum persist effect, set `holdRef.current=true` sinkron sebelum `setRestorable`, dan persist effect early-return `if (!bootRef.current || holdRef.current)`. Refs (bukan state) supaya dibaca sinkron tanpa churn re-render.
- Restore WAJIB sanitasi bentuk anggota dari JSON (role di-whitelist, field string aman, clamp maxSpecialists 1тАУ5) тАФ localStorage bisa rusak/diutak-atik.
- Draft dihapus saat create sukses & reset(); semua akses localStorage dibungkus try/catch.

## Tahap 32 тАФ Tim bertingkat / berjenjang (nested teams) (done)
Wizard `/organization-builder` kini bisa membangun struktur >2 level (pohon), bukan cuma star datar 1-lead. Field opsional `parentLocalId` per anggota; sumber kebenaran tunggal = helper murni `computeOrgTree(members)`. Murni klien, backward-compatible.
- **Satu helper murni jadi sumber kebenaran topologi.** `computeOrgTree(members)` menurunkan: akar (orchestrator tanpa atasan eksplisit valid = Ketua puncak), `validExplicitParent` (guard: ada + role orchestrator + bukan diri + tanpa siklus), `effectiveParentOf` (non-akar tanpa atasan eksplisit тЖТ default akar = kompat. datar), `effectiveChildrenOf` (bagan + buildOrg), `explicitDescendants` (filter opsi anti-siklus). buildOrg + validasi + UI + bagan SEMUA baca dari helper ini тАФ jangan duplikasi logika atasan/akar di tempat lain. **Why:** dulu wiring (leadтЖТall), validasi (orchestratorCount===1), dan chart (lead+others) tiga sumber terpisah; menambah jenjang tanpa menyatukan = drift. **How to apply:** perubahan aturan hierarki cukup di `computeOrgTree`.
- **Atasan WAJIB orchestrator; akar = orchestrator tanpa atasan valid.** Validasi berubah dari `orchestratorCount===1` тЖТ `rootCount===1`: banyak Ketua Tim kini BOLEH (sebagai sub-pemimpin), yang wajib tepat satu adalah AKAR. **Why:** jenjang butuh >1 orchestrator. **How to apply:** jangan kembalikan gating ke orchestratorCount; pakai rootCount.
- **buildOrg edge berbasis atasan efektif** (`fromLocalId=effectiveParentOf(m)`, selalu orchestrator), `leadLocalId`=akar. Cocok dgn lint schema `structure.edges` (fromLocalId harus orchestrator) тЖТ nesting sudah didukung engine/runtime, wizard dulu cuma belum memakainya.
- **Cegah siklus di TIGA lapis:** (1) `validExplicitParent` guard rantai ke atas, (2) opsi Select "Lapor ke" mengecualikan `explicitDescendants(self)`, (3) `effectiveParentOf` fallback aman. Atasan menggantung (anggota dihapus / role atasan diturunkan jadi specialist) di-handle: `computeOrgTree` abaikan parent tak valid тЖТ anak reattach ke akar; `removeMember` bersihkan ref; `sanitizeDraft` strip parent tak dikenal; `duplicateMember` bawa parentLocalId.
- testid: `select-parent-{localId}` (Select "Lapor ke", gate `members.length>1`), `field-parent-{localId}`, bagan rekursif tetap `chart-lead` (akar) + `chart-node-{localId}`. Opsi sentinel "Ketua puncak" = konstanta `ROOT_SENTINEL` ("__root__") тЖТ dipetakan ke `parentLocalId: undefined`.

## Tahap 33 тАФ Gerbang Manusia (тЧЖ) per anggota (done)
Tiap anggota di `/organization-builder` bisa mendeklarasikan `gates: string[]` opsional = keputusan yang TIDAK boleh diambil agen sendiri (mewujudkan elemen inti ebook Buku II Bab 3: "alur tanpa gerbang adalah pelarian"). Murni klien; tanpa perubahan schema/engine/API.
- **Injeksi gates ke systemPrompt HANYA saat `buildOrg(fillDefaults=true)`** (jalur analyze/configure nyata). Saat `fillDefaults=false` (infer "Sempurnakan Detail" + regenerate) systemPrompt tetap raw, JANGAN sisipkan `gateBlock`. **Why:** infer/regenerate mengandalkan systemPrompt kosong untuk tahu field mana yang harus dibuatkan; menyisipkan blok gerbang ke prompt kosong membuatnya "tak-kosong" тЖТ inferensi melewatkannya. **How to apply:** efek samping presentasional/instruksional apa pun yang ditambahkan ke prompt anggota harus di-gate `fillDefaults` agar tidak mencemari deteksi field-kosong.
- **`sanitizeGates(gates)`** (trim, buang kosong, cap 160 char/item & 8 item) dipanggil DI DALAM `sanitizeDraft` тЖТ gates otomatis ikut alur draft localStorage (Tahap 29) & ekspor/impor JSON (Tahap 30) tanpa kode tambahan. `gateBlock(gates)` juga lewat sanitizeGates sebelum merangkai blok "GERBANG MANUSIA (тЧЖ) тАФ WAJIB".
- **Konkatenasi string gate ke systemPrompt aman** тАФ gates hanya jadi teks instruksi LLM (bukan HTML/script sink), tak ada XSS. `duplicateMember` salin gates sbg array baru; `hasMeaningfulDraft` kini kenali draft gate-only.
- testid: `field-gates-{localId}`, `input-gate-{localId}-{i}`, `btn-add-gate-{localId}`, `btn-remove-gate-{localId}-{i}`, badge тЧЖ `gate-badge-{localId}` di bagan.
- **Gerbang manusia menciut ke pemilik untuk tim kecil/sumber daya terbatas.** Prinsipnya soal SIAPA pemegang pertimbangan akhir, bukan jumlah orang; di kontraktor 1-2 orang (mis. suami-istri, hanya satu aktif) si pemilik = Ketua = pemegang gerbang sekaligus.
